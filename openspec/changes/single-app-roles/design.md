## 项目背景

原方案使用两个独立的微信小程序，孩子版和管理员版分开维护。现调整为单一小程序，通过用户角色（child/admin）动态区分功能，减少维护成本。同时预设超级管理员账号，确保系统可被管理。

## 目标 / 非目标

**目标：**
- 单一微信小程序，孩子和管理员使用同一个小程序
- 根据用户角色动态显示不同导航和功能
- 预设超级管理员账号 admin，默认密码 123456
- 首次登录必须修改密码

**非目标：**
- 不支持用户自行注册为管理员
- 不支持修改管理员账户

## 技术决策

### 用户角色权限模型

**决策：** 三级角色模型

**角色定义：**
- `super_admin`：超级管理员（预设），可管理所有功能
- `admin`：普通管理员（由超级管理员创建）
- `child`：孩子用户（通过微信登录）

### 密码存储方案

**决策：** 密码使用 bcrypt 哈希存储

**理由：**
- bcrypt 是成熟的密码哈希算法
- 带盐值，防止彩虹表攻击
- 可配置工作因子的计算成本

**数据库字段调整：**
```sql
ALTER TABLE users ADD COLUMN password_hash VARCHAR(255);
ALTER TABLE users ADD COLUMN is_first_login BOOLEAN DEFAULT TRUE;
```

### 角色导航设计

**决策：** 登录后根据角色动态加载不同页面配置

**孩子用户导航：**
- 首页（积分余额）
- 记录奖励
- 奖励历史
- 奖品目录
- 兑换记录

**管理员导航：**
- 仪表盘（待审核数量）
- 奖励审核
- 兑换审核
- 奖品管理

### 强制密码修改流程

**决策：** 首次登录时拦截并强制跳转修改密码页面

**流程：**
1. 用户登录成功
2. 检查 `is_first_login` 标志
3. 为 true 时跳转到修改密码页面
4. 允许跳过首页、功能页访问
5. 修改密码成功后重置标志

## 风险与权衡

**风险：** 忘记密码无法登录
**缓解：** 超级管理员可以重置其他管理员密码

**风险：** 密码强度不够
**缓解：** 要求密码至少6位，包含字母和数字

**风险：** 首次登录时修改密码页面可能被绕过
**缓解：** 所有 API 检查 `is_first_login` 标志，未修改密码的请求返回错误
