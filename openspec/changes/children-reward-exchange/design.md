## 项目背景

这是一个新的微信小程序系统，包含两个角色：孩子（获取奖励/兑换奖品）和管理员（审核申请）。数据存储在 CloudBase MySQL 中。系统需要支持两个小程序共享同一个数据库，但根据用户角色提供不同功能。

## 目标 / 非目标

**目标：**
- 孩子可以使用微信 OpenID 注册/登录
- 孩子可以提交奖励申请及描述
- 孩子可以浏览奖品目录并提交兑换申请
- 管理员可以查看所有待审核申请并筛选
- 管理员可以批准/拒绝申请并留言
- 微信分享功能方便管理员快速审批

**非目标：**
- 实时推送通知（使用 CloudBase 服务通知）
- 家长仪表盘（后续功能）
- 奖励数据分析/报表（后续功能）

## 技术决策

### 架构：独立小程序 vs 单一小程序

**决策：** 使用单一小程序，根据角色导航

**理由：** 更容易维护，共享数据库，部署简单。孩子看到首页→奖励→奖品流程；管理员看到仪表盘→审核流程。角色在登录时确定。

### 数据库：CloudBase MySQL 表

**决策：** 使用关系型数据库存储结构化数据

**理由：** MySQL 提供更好的查询能力：
- 按状态、日期、用户筛选申请
- 积分余额更新的事务完整性
- 管理员仪表盘的复杂查询

**数据表：**
- `users`: id, openid, nickname, role (child/admin), points_balance, created_at
- `prizes`: id, name, description, points_cost, inventory, image_url, is_active
- `reward_applications`: id, user_id, description, status, points, admin_note, created_at, reviewed_at
- `redemption_applications`: id, user_id, prize_id, status, admin_note, created_at, reviewed_at

### 认证：微信登录 + 自定义注册

**决策：** 使用微信登录获取 OpenID，昵称自定义设置

**理由：** 微信提供安全 OpenID，无需完整 OAuth。孩子首次登录后设置昵称。管理员在数据库中预创建。

### 角色权限控制

**决策：** 角色存储在用户记录中，在 API 层面检查

**理由：** 简单的 RBAC。孩子 API 拒绝管理员角色，管理员 API 需要管理员角色。MVP 不需要复杂 ACL。

### 分享给管理员流程

**决策：** 生成带申请 ID 参数的可分享页面

**理由：** 微信分享 API 允许自定义分享标题/路径。管理员打开分享链接 → 自动跳转到审核页面并预加载申请 → 一键批准/拒绝。

## 风险与权衡

**风险：** 扣除积分时的竞态条件
**缓解：** 使用数据库事务和行级锁（SELECT FOR UPDATE）

**风险：** 孩子可能将管理员链接分享给他人
**缓解：** 管理员页面检查登录角色；非管理员用户跳转到首页

**风险：** 管理员拒绝后孩子已消费积分导致积分不一致
**缓解：** 仅在管理员批准后才扣除积分；兑换状态变为已批准 → 原子性扣除积分

## 部署计划

1. 创建 CloudBase MySQL 数据库和数据表
2. 部署云函数实现所有 API
3. 上传孩子小程序
4. 上传管理员小程序
5. 手动创建初始管理员账号
