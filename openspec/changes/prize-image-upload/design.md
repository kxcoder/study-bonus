## 项目背景

现有系统使用 CloudBase MySQL 存储数据，未来可能需要迁移到其他云厂商。需要在不修改业务逻辑的情况下，支持底层数据库的切换。同时需要为管理员小程序新增奖品图片上传功能。

## 目标 / 非目标

**目标：**
- 设计数据库抽象层，支持多云迁移
- 实现奖品图片上传功能，图片压缩到1KB以内
- 图片数据以 Base64 格式存储在数据库

**非目标：**
- 不支持其他类型文件上传（仅图片）
- 不实现图片 CDN 加速（后续功能）

## 技术决策

### 数据库抽象层设计

**决策：** 使用数据访问层（DAL）模式封装数据库操作

**理由：** 将业务逻辑与数据库操作解耦，通过接口抽象数据库操作。未来迁移时只需实现新的数据库适配器，无需修改业务代码。

**架构：**
```
业务层 → 数据访问接口 → 数据库适配器 → CloudBase MySQL
                         ↓
                    其他数据库适配器（未来）
```

### 图片存储方案

**决策：** 图片压缩后以 Base64 格式存储在数据库 BLOB/TEXT 字段

**理由：**
- 简化架构，无需额外对象存储服务
- 1KB 大小对数据库性能影响可控
- 支持快速读取和展示

**备选方案：**
- 使用对象存储（COS/AWS S3）：需要额外配置，不符合当前需求
- 使用文件服务器：需要自建，后续维护成本高

### 图片压缩方案

**决策：** 在小程序端使用 canvas 压缩

**理由：** 微信小程序提供 canvas 2D API，可以在客户端完成压缩，减少服务端压力。

**压缩算法：**
1. 使用 canvas 绘制图片
2. 按比例缩放（最长边不超过100像素）
3. 导出为 JPEG 格式，质量设为 0.3 ~ 0.5
4. 循环降低质量直到文件大小 < 1KB

### 数据库表调整

**决策：** prizes 表新增 image 字段

**字段定义：**
```sql
ALTER TABLE prizes ADD COLUMN image TEXT COMMENT '奖品图片Base64编码';
```

## 风险与权衡

**风险：** Base64 编码增加 33% 存储大小
**缓解：** 控制图片压缩在 1KB 以内，实际存储约 1.3KB

**风险：** 大图片影响数据库查询性能
**缓解：** 图片独立查询，不与其他字段一起查询

**风险：** 多云迁移时的兼容性问题
**缓解：** 通过抽象层隔离，业务代码不直接依赖 SQL
